To understand why the app feels "live" without a database or file changes, we have to look at how React manages Memory (RAM) versus Storage (Disk).

1. The "Once and Only Once" Reading
When your app starts, it executes the code in AppContext.js. You’ll notice this line:

javascript
import initialStock from '../data/stock.json';
At this exact moment, the JavaScript engine reads the stock.json file from your disk and converts it into a JavaScript Object in your phone's RAM.

Crucial Point: From this second onwards, the app never looks at the .json file again. It only works with the copy it has in memory. This is why the .json file on your disk remains unchanged even if you "buy" every item in the app.

2. How setState Actually Works (The Engine Room)
When you call setProductReviews(newArray), you aren't just changing a variable. You are triggering a complex process called Reconciliation.

A. Immutability (The [...] Syntax)
In React, you never do state.push(item). You always do setState([...state, item]). Why? React uses "Shallow Comparison." It looks at the "memory address" of your state. If you just push to an array, the memory address stays the same, and React thinks nothing changed. By using [...] (the spread operator), you create a brand new array in a new memory slot. React sees the new slot and says: "Aha! Something is different. I must re-render."

B. The "Subscriber" Model
Because productReviews is inside AppContext.Provider, every screen that uses useAppContext() is a subscriber.

You call submitReview in ReviewScreen.
setState updates the global reviews array in RAM.
React immediately "pings" every component that is currently mounted and watching that state (like the ProductScreen).
3. The "Live Update" Illusion
How does the rating update without a reload? This is due to Computed Properties.

In ProductScreen.js, we don't store the avgRating in a separate state. Instead, we calculate it during the render:

javascript
const thisProductReviews = productReviews.filter(r => r.productId === product.id);
const avgRating = calculateAverage(thisProductReviews); // Done every time the component renders
The Chain Reaction:

User clicks "Submit Review".
Global State productReviews changes.
React triggers a re-render of ProductScreen.
The code for ProductScreen runs again from top to bottom.
It sees the updated productReviews array, calculates a new avgRating, and generates a new UI.
React’s Virtual DOM compares the old UI with the new UI and only changes the tiny piece of text showing the rating on the screen.
4. Why no database is needed?
In a production app, the database is just a way to make sure that when you close the app and reopen it, the data is still there. In your current "Audit" setup:

Live Updates: Handled by React's state in RAM.
Persistence: None. If you refresh the app or close it, all your reviews and cart items disappear, and the app "re-boots" by reading the original stock.json again.
Summary of the "Mental Model"
Think of stock.json as a Seed. You plant it in the App's memory (RAM) when the app wakes up. Your useState is the Water that changes the plant. You can grow the plant as much as you want in the memory, but the Seed in your drawer (the file on disk) remains exactly as it was.
